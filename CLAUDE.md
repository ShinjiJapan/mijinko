# 指示
 - serena MCPを使い概要を随時確認。ソースを更新したらserenaも更新
 - ソースコードを修正したら対象プロジェクトをdeployし、deployしたと報告する
 - deploy時は配布用の単一exeも再発行する。発行前に実行中の`Filer.App`を停止(単一exeロック対策)してから:
   `dotnet publish src/Filer.App/Filer.App.csproj -c Release -r win-x64 --self-contained -p:PublishSingleFile=true -o publish` → `publish/Filer.App.exe`
 - その後必要があればブラウザーでの確認(browser-testスキル)まで実施
 - ユーザーへの問いかけや回答は必ず日本語で


# ドキュメント
 - ドキュメント作成を依頼されたらmemoフォルダー配下にmermaid/markdownで保存
 - 特に指定されなければ新規ファイルとして作成
 - ドキュメント修正時に履歴を残さない.最新状態だけのドキュメントにする

# 禁止事項
 - 「調査して」などと依頼されたら調査し結果を報告するのみ。勝手にコードの修正はしないこと
 - 勝手にフォールバックを実装しない。フォールバックはバグを隠蔽し有害です。特にバグ修正時にバグの根本原因を明らかにせず「とりあえずフォールバック」は厳禁
 - 修正時に互換性を残す場合はユーザーに確認する
 - コメントは最新の状態についてのコメントのみ。履歴不要

# 処理完了後
 - 実施した内容で今後も使うようなことはスキル化が妥当であればスキル化する
 - スキルやエージェントなどでスムーズにいかなかった部分で改善可能な部分はスキル・エージェントの修正を行う

# 調査依頼されたら
 - ソースコードだけでなくログも必要に応じて調査。
 - 調べればわかることを憶測で回答しないこと
 - ログが不十分なため調査に支障があった場合はログ出力するコードを追加する
# 実装
 - TDDで実装する。
 - 実装完了後コードレビューする
 - 修正したファイル一覧をフルパスで提示する

# UI(ダイアログ)
 - ダイアログにボタンを追加する場合は Styles\Controls.xaml の共通スタイル `DialogButton` を使う
 - 各ダイアログの Window.Resources に `<Style TargetType="Button" BasedOn="{StaticResource DialogButton}"/>` を置き、テーマ追従(角丸・ホバー・無効表現)を揃える
 - 独自のボタンテンプレートを各ダイアログに直書きしない

# ファイル作成
 - 実装以外でrootフォルダに勝手にファイルを追加しない
 - 基本は agent配下に置く。使い捨ての一時ファイルはagent\tmp 配下に置く

 # 動作確認
 - SendKeys ではなく UI Automation の InvokePattern 等でフォーカスを奪わずに行えるのであればそのように(その際、最前面化せずFilerにキー送信できるのであれば最前面化は不要)
 - SendKeys等、フォーカスが必要なキー操作の場合はAttachThreadInputでFilerを最前面にした後で実施
